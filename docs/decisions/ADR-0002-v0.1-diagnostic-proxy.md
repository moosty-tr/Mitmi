# ADR-0002: v0.1 Diagnostic Proxy Mode

## Status

Accepted for planning.

## Context

MITMI's long-term vision includes observation, interception, logging, replay, fault simulation, caching, queuing, bridging, metrics, and automation.

For v0.1, the product needs a narrower center of gravity. The accepted direction is to prove that MITMI can run as a reliable Modbus TCP diagnostic proxy before adding active intervention features.

## Decision

MITMI v0.1 will be an explicit in-path diagnostic proxy for Modbus TCP.

Clients connect to MITMI. MITMI connects to the configured upstream Modbus TCP server. MITMI forwards traffic unchanged while recording Modbus-aware diagnostics, logs, captures, and metrics.

v0.1 should be Modbus-aware from day one, but it should not modify traffic by default.

## Consequences

Positive:

- The first product experience is easy to explain and test.
- The system proves real TCP forwarding behavior early.
- The Modbus plugin boundary is tested with a practical protocol.
- Logging and capture become useful before replay and fault simulation are attempted.

Negative:

- v0.1 is not a true passive sniffer.
- Existing clients must be configured to connect through MITMI.
- MITMI becomes part of the communication path, so reliability and latency matter immediately.
- The diagnostic proxy path may expose limitations in generic Modbus libraries.

## Guardrails

- Keep active traffic modification out of v0.1 unless a later decision explicitly changes scope.
- Preserve raw traffic even when Modbus decoding fails.
- Keep Modbus-specific behavior inside the Modbus protocol project.
- Make pass-through reliability more important than perfect decoding.
- Treat replay as dependent on trustworthy capture records, not as the first implementation slice.

## Alternatives Considered

### Passive Sniffer First

Rejected for v0.1. It would require packet capture support, platform-specific behavior, elevated permissions, and more complex stream reconstruction.

### Replay Tool First

Rejected for v0.1. Replay quality depends on a solid capture model and request/response correlation. Building replay first would likely create false confidence.

### Fault Simulation First

Rejected for v0.1. Fault simulation is powerful but potentially unsafe. It should be added only after pass-through diagnostics are reliable and explainable.
