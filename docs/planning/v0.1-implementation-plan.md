# v0.1 Implementation Plan

## Status

Planning draft.

This document turns the accepted v0.1 architecture and ADRs into an implementation sequence. It does not approve coding by itself; it defines what should be built first once implementation is approved.

## 1. Planning Inputs

This plan follows these accepted directions:

- v0.1 is an explicit in-path Modbus TCP diagnostic proxy.
- v0.1 supports one diagnostic session.
- Clients must be retargeted to MITMI's listen endpoint.
- Forwarding is latency-first and diagnostics are asynchronous where practical.
- Logging and capture are separate outputs.
- Captures are enabled by default, with visible output path and retention/sensitivity warnings.
- NModbus remains isolated to `Mitmi.Protocols.Modbus`.
- Replay, fault simulation, caching, queuing, bridging, dynamic plugin loading, service installation, and a web dashboard are out of the first implementation path.

## 2. Recommended First Project Shape

Start with four production projects:

- `Mitmi.Domain`
- `Mitmi.Application`
- `Mitmi.Protocols.Modbus`
- `Mitmi.Host.Console`

Use test projects only where the current slice needs them:

- `Mitmi.Application.Tests`
- `Mitmi.Protocols.Modbus.Tests`
- `Mitmi.IntegrationTests`

Recommended reference direction:

```text
Mitmi.Host.Console
  -> Mitmi.Application
  -> Mitmi.Protocols.Modbus

Mitmi.Protocols.Modbus
  -> Mitmi.Application
  -> Mitmi.Domain
  -> NModbus

Mitmi.Application
  -> Mitmi.Domain

Mitmi.Domain
  -> no MITMI project dependencies
```

Do not create `Mitmi.Abstractions` in the first slice. A separate contracts assembly is attractive, but it is premature before the first host/application/plugin boundary is exercised by real code. If `Mitmi.Application` starts accumulating contracts that are useful without orchestration, split them later with evidence.

Do not create `Mitmi.Infrastructure` in the first slice. Add it when file logging, capture storage, metrics adapters, or configuration adapters become large enough that leaving them in the host would blur the composition root.

## 3. Project Responsibilities

### `Mitmi.Domain`

Owns protocol-neutral concepts:

- Session identifier.
- Connection identifier.
- Endpoint value objects.
- Traffic direction.
- Message or traffic envelope.
- Correlation identifier.
- Pipeline decision.
- Diagnostic warning.
- Capture record shape at the conceptual level.

No Modbus names, NModbus types, logging framework types, file paths, host lifetime types, or JSON serialization attributes should live here unless there is a strong reason.

### `Mitmi.Application`

Owns orchestration and stable inward-facing contracts:

- Validated runtime configuration model.
- Configuration validation rules.
- Session orchestration.
- Protocol plugin registry.
- Protocol frame decoder contracts.
- Diagnostic event contracts.
- Capture and metrics ports.
- Bounded asynchronous diagnostics policy.
- Shutdown coordination.

It can know that a session has a protocol, listen endpoint, upstream endpoint, logging settings, capture settings, and diagnostic options. It should not know Modbus function codes.

### `Mitmi.Protocols.Modbus`

Owns Modbus-specific behavior:

- Plugin registration for `modbus-tcp`.
- MBAP header parsing.
- Function code extraction.
- Exception response detection.
- Request/response correlation metadata.
- Modbus-specific validation warnings.
- NModbus integration where useful.

The first parser should be a small MITMI-owned Modbus TCP frame detector. NModbus should assist semantics and future test/replay support; it should not own the raw forwarding or capture model.

### `Mitmi.Host.Console`

Owns process concerns:

- Command-line arguments.
- Configuration file loading.
- Path resolution relative to the configuration file.
- Dependency composition.
- Console lifetime.
- Startup diagnostics.
- Exit codes.

The host is the composition root, not the application layer.

## 4. Configuration Shape

Use a single JSON file with a singular `session` object:

```json
{
  "configurationVersion": 1,
  "logging": {
    "console": {
      "enabled": true,
      "minimumLevel": "Info"
    },
    "file": {
      "enabled": true,
      "minimumLevel": "Info",
      "path": "./logs/mitmi.log"
    }
  },
  "capture": {
    "enabled": true,
    "outputPath": "./captures",
    "retention": {
      "mode": "Manual"
    }
  },
  "metrics": {
    "enabled": true,
    "sink": "Log"
  },
  "session": {
    "id": "default",
    "protocol": "modbus-tcp",
    "listenEndpoint": {
      "address": "0.0.0.0",
      "port": 1502
    },
    "upstreamEndpoint": {
      "address": "127.0.0.1",
      "port": 502
    },
    "diagnostics": {
      "decodeProtocol": true,
      "captureRawPayloads": true
    }
  }
}
```

Configuration choices:

- Use `configurationVersion`, not a generic `version`, because capture records and other future artifacts will also have versions.
- Resolve relative file paths relative to the configuration file directory, not the process working directory.
- Keep `session`, not `sessions`, for v0.1.
- Keep endpoint names explicit: `listenEndpoint` and `upstreamEndpoint`.
- Keep console and file logging enabled by default while making explicit disabling possible.
- Allow DNS names for upstream endpoints, but prefer IP addresses or wildcard addresses for listen endpoints.
- Keep protocol-specific options under `session.diagnostics` only while they remain generic. Add a `protocolOptions` object later when Modbus-specific options appear.

## 5. Configuration Validation Rules

Validation must complete before any TCP listener starts.

Minimum errors:

- Missing configuration file.
- Unsupported `configurationVersion`.
- Missing `session`.
- Empty or duplicate-reserved `session.id`.
- Unknown `session.protocol`.
- Invalid listen address.
- Invalid listen port.
- Missing upstream address.
- Invalid upstream port.
- Missing logging configuration.
- Invalid logging level.
- File logging enabled without a file path.
- Capture enabled without an output path.
- Unknown capture retention mode.

Minimum warnings:

- Capture enabled with manual retention.
- Capture enabled because captures may contain sensitive industrial data.
- Listen port is below 1024 on Unix-like systems.
- Listen endpoint and upstream endpoint appear identical.
- Relative output paths were resolved to absolute paths.
- File logging uses `Debug`.
- Console logging uses `Debug`.

The validation result should be structured, not a bag of strings. The host can render it for humans, but tests should assert error codes.

## 6. Startup Diagnostics

At startup, MITMI should print or log:

- Configuration file path.
- Effective session ID.
- Protocol ID.
- Listen endpoint.
- Upstream endpoint.
- Console minimum log level.
- File minimum log level and resolved file path.
- Capture enabled/disabled.
- Resolved capture output path when enabled.
- Capture retention warning when enabled.
- Capture sensitivity warning when enabled.
- Explicit client-retargeting reminder.

Do not put raw traffic payloads in startup or normal `Info` logs.

## 7. First Runnable Slice

The first implementation slice should be a runnable console host that validates configuration and exits predictably.

Scope:

- Add `Mitmi.Domain`, `Mitmi.Application`, and `Mitmi.Host.Console`.
- Keep the existing `Mitmi.Protocols.Modbus` project.
- Add project references according to the graph above.
- Add a minimal static registration path for the Modbus plugin.
- Load a JSON configuration file from `--config <path>`.
- Support `--validate-config` to validate and exit without starting networking.
- Validate the configuration into an application runtime model.
- Resolve output paths relative to the configuration file.
- Render startup diagnostics and warnings.
- Return nonzero exit code for validation errors.

Out of scope:

- Opening a TCP listener.
- Connecting to upstream.
- Forwarding traffic.
- Decoding Modbus frames.
- Writing log files.
- Writing captures.
- Metrics export.
- Replay.

This slice is intentionally humble. It proves the project boundaries, configuration contract, path resolution rule, plugin registration direction, and validation diagnostics before network code makes mistakes harder to isolate.

## 8. Second Runnable Slice

The second implementation slice should create the first actual diagnostic proxy path.

Scope:

- Start one TCP listener from the validated session.
- Accept one client connection at a time unless explicitly expanded later.
- Open one upstream TCP connection per accepted client.
- Forward bytes bidirectionally without modification.
- Shut down both directions predictably when either side closes.
- Log connection lifecycle events.
- Preserve cancellation and graceful shutdown behavior.

Out of scope:

- Modbus frame decoding.
- Capture writing.
- Replay.
- Multiple sessions.
- Traffic modification.

This is the first product-behavior slice. It should have integration tests using local TCP endpoints before Modbus semantics are added.

## 9. Follow-On Slices

Recommended sequence after pass-through works:

1. Add bounded asynchronous diagnostic event channel.
2. Add Modbus TCP frame detection in `Mitmi.Protocols.Modbus`.
3. Add request/response correlation metadata.
4. Add structured console and file logging with independent minimum levels.
5. Add versioned append-only capture records.
6. Add basic local metrics emitted through logs or an in-process sink.
7. Add integration tests with a local Modbus TCP client/server setup.

Do not start replay until capture records are trustworthy enough to be used as evidence.

## 10. Capture Format Direction

Use versioned append-only NDJSON for the first capture store unless performance evidence says otherwise.

Rationale:

- It is easy to inspect in support scenarios.
- It is stream-friendly.
- It avoids mixing replay data into normal logs.
- It can preserve raw payloads as base64.
- It can evolve through an explicit capture format version.

Early capture records should include:

- Capture format version.
- Timestamp in UTC.
- Session ID.
- Connection ID.
- Direction.
- Protocol ID.
- Raw payload as base64 when enabled.
- Correlation ID when available.
- Protocol metadata object when available.
- Decode warnings.

This format should be documented as experimental for v0.1 planning. A future binary or indexed store can be introduced after the capture model is proven.

## 11. Diagnostics Queue Policy

Latency-first forwarding requires bounded diagnostics.

Recommended initial policy:

- Forwarding writes do not wait for diagnostic consumers.
- Diagnostic events go through a bounded channel.
- If the channel is full, drop the newest diagnostic event and increment a loss counter.
- Emit a visible warning when loss starts and summarize total loss during shutdown.

This favors traffic continuity over diagnostic completeness, matching ADR-0008. A later full-fidelity capture mode can choose a different policy explicitly.

## 12. Testing Plan

First slice tests:

- Valid configuration becomes a validated runtime model.
- Missing sections produce stable error codes.
- Bad endpoints fail before networking.
- Relative paths resolve relative to the config file directory.
- Capture warnings appear when capture is enabled.
- Unknown protocol fails validation.
- Modbus plugin registration makes `modbus-tcp` known without leaking NModbus types.

Second slice tests:

- Local TCP client traffic reaches local upstream server unchanged.
- Upstream response reaches client unchanged.
- Upstream connect failure is reported.
- Listener bind failure is reported.
- Closing client or upstream tears down the paired connection predictably.
- Cancellation stops the listener and active forwarding loops.

Modbus slice tests:

- Complete Modbus TCP ADUs are detected.
- Split TCP reads are reassembled for diagnostics.
- Malformed frames produce warnings but do not block raw forwarding.
- Transaction identifiers correlate request/response metadata.
- Exception responses are identified.

## 13. Dependencies To Decide During Coding

Default choices unless contradicted by implementation evidence:

- Use `System.Text.Json` for configuration.
- Use built-in `TcpListener`, `TcpClient`, and `NetworkStream` for the first TCP proxy.
- Use .NET cancellation tokens for shutdown.
- Avoid `System.CommandLine` until the CLI has more than a few stable commands.
- Keep NModbus only in `Mitmi.Protocols.Modbus`.

Logging dependency remains an implementation choice. If independent console/file levels become awkward with the built-in providers, use a mature logging provider in the host only. Do not build a custom general-purpose logging framework.

## 14. Explicit Deferrals

Defer these until the diagnostic proxy foundation works:

- Dynamic plugin loading.
- Separate `Mitmi.Abstractions` project.
- Separate `Mitmi.Infrastructure` project.
- Windows Service or Linux daemon support.
- Transparent proxying.
- Passive packet capture.
- Replay.
- Fault simulation.
- Caching.
- Queuing.
- Bridging.
- Web dashboard.
- Licensing enforcement.

The important architectural point is to preserve places where these can attach later, not to prebuild the full platform before the first proxy run.

## 15. Approval Gate

Implementation should start only after this plan is accepted or revised.

Recommended first coding task:

Create the project structure and configuration validation slice described in section 7.
