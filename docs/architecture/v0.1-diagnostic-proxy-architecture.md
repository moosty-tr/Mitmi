# v0.1 Diagnostic Proxy Architecture

## Status

Planning draft.

This document describes the recommended v0.1 architecture direction. It is not an implementation specification yet.

## 1. Product Position For v0.1

MITMI v0.1 should be an explicit in-path Modbus TCP diagnostic proxy.

The first version should answer one practical field question well:

"What is happening between this Modbus TCP client and server, and can I record enough context to diagnose it?"

v0.1 should prioritize:

- Reliable pass-through behavior.
- One diagnostic session only.
- Modbus-aware request/response diagnostics.
- Structured logs.
- Replay-oriented capture foundations.
- Clear startup and runtime diagnostics.
- Cross-platform console operation on Windows and Linux.
- Lightweight engineer laptop deployment.
- Explicit client retargeting to MITMI's IP/port.
- Minimum-latency forwarding with asynchronous diagnostics.

v0.1 should not prioritize:

- Transparent packet sniffing.
- Network tap mode.
- Traffic modification.
- Fault simulation.
- Caching.
- Queuing.
- Bridging.
- Dynamic plugin loading.
- Web dashboard.
- Service or daemon installation.
- Production gateway deployment.
- Transparent proxying, NAT, ARP spoofing, or DNS spoofing.

## 2. Important Constraint: Proxy Is Not Sniffer

The word "observe" can mean two different things:

- In-path observation: MITMI accepts client connections and forwards traffic to the configured upstream server.
- Passive sniffing: MITMI watches traffic on a network interface without participating in the connection.

v0.1 should use in-path observation.

Reason:

- It is easier to implement safely and portably in .NET.
- It avoids packet capture drivers and elevated OS permissions.
- It works consistently on Windows and Linux.
- It gives MITMI full access to ordered TCP streams.
- It fits the future middleware pipeline model.

Trade-off:

- Existing clients must be pointed at MITMI instead of the original server.
- MITMI becomes part of the communication path, so reliability and latency matter immediately.
- Some production environments may not allow inserting a proxy into the path.

Future passive sniffing should be treated as a separate capability with separate architecture, platform support, and licensing implications.

## 3. Deployment Target

v0.1 should target engineer laptops first.

This means:

- Run as a console application.
- Avoid requiring Windows Service or Linux daemon installation.
- Avoid requiring elevated packet-capture permissions.
- Prefer simple local configuration files.
- Prefer visible local log and capture output.
- Make startup diagnostics very clear.

Expected laptop-specific risks:

- Local firewall blocks the listen port.
- The chosen port is already in use.
- VPN, Wi-Fi, Ethernet adapter changes, or laptop sleep interrupt traffic.
- Corporate endpoint protection interferes with local proxy behavior.
- Working-directory-relative paths confuse users.

Mitigation:

MITMI should report resolved endpoints and output paths clearly at startup. Documentation should call out firewall, port, and network-adapter considerations.

## 4. Network Placement

v0.1 requires explicit client retargeting.

The user must configure the Modbus TCP client, simulator, SCADA test instance, or other initiating system to connect to MITMI's configured IP address and port. MITMI then connects to the configured upstream Modbus TCP server.

v0.1 should not attempt:

- Transparent proxying.
- NAT or router-style interception.
- ARP spoofing.
- DNS spoofing.
- Passive packet capture.

Trade-off:

Explicit retargeting is less seamless than transparent interception, but it is simpler, safer, easier to explain, easier to test, and better aligned with engineer-laptop deployment.

## 5. Recommended Runtime Shape

The v0.1 runtime should have these conceptual components:

- Console host.
- Configuration loader and validator.
- Single-session orchestrator.
- Protocol registry.
- Modbus TCP protocol plugin.
- TCP listener.
- Upstream connector.
- Bidirectional forwarding loop.
- Message framing and correlation layer.
- Middleware pipeline.
- Structured logging sink.
- Console log output.
- File log output.
- Capture sink.
- Metrics sink.
- Shutdown and diagnostics coordinator.

The central operational unit should be a session.

A session represents one configured diagnostic proxy instance:

- Local listen endpoint.
- Upstream target endpoint.
- Protocol.
- Pipeline.
- Logging and capture settings.
- Runtime state.

Recommendation: use a single `session` as the user-facing configuration concept for v0.1. Internally, a session may contain one route from local listener to upstream target.

v0.1 should not expose multi-session behavior. If a user needs multiple independent diagnostic paths, the recommended workaround is to run multiple MITMI processes with separate configuration files and ports.

## 6. Data Flow

The basic v0.1 flow should be:

1. MITMI starts from a JSON configuration file.
2. Configuration is validated before any listener starts.
3. The Modbus plugin is registered.
4. One diagnostic session is created.
5. The session opens one local TCP listener.
6. A Modbus client that has been explicitly configured to use MITMI connects to MITMI.
7. MITMI opens an upstream TCP connection to the configured Modbus server.
8. Bytes are forwarded in both directions.
9. The Modbus plugin frames Modbus TCP Application Data Units where possible.
10. MITMI correlates request and response traffic using transaction metadata.
11. Logs, capture records, and metrics are emitted.
12. Shutdown flushes outputs and reports session summaries.

Key principle: forwarding must not depend on successful semantic decoding. If decoding fails, MITMI should still be able to forward raw traffic unless policy says otherwise.

Forwarding should also not depend on successful log writes, capture writes, or metrics export. These diagnostics should run asynchronously wherever practical.

## 7. Modbus Awareness In v0.1

Modbus-aware does not mean Modbus-dominates-the-core.

The Modbus plugin should understand enough to support diagnostics:

- MBAP header.
- Transaction identifier.
- Protocol identifier.
- Length.
- Unit identifier.
- Function code.
- Request/response direction.
- Exception response detection.
- Basic request/response correlation.
- Raw frame preservation.

Optional if low-risk:

- Address and quantity extraction for common read/write function codes.
- Human-readable operation names.
- Basic validation warnings.

Should not be required in v0.1:

- Full Modbus semantic validation.
- Device simulation.
- Response generation.
- Write rewriting.
- Advanced replay editing.
- Modbus RTU.

## 8. NModbus Boundary

NModbus is accepted as the initial Modbus dependency, but it should not define MITMI's architecture.

Recommended usage:

- Keep NModbus referenced only by the Modbus protocol project.
- Use it for protocol semantics, validation helpers, future test endpoints, and possibly replay support.
- Keep MITMI-owned raw frame preservation and correlation.
- Do not expose NModbus types from core abstractions.

Risk:

NModbus is primarily a Modbus client/server library. A diagnostic proxy needs low-level visibility into traffic moving through it. If MITMI delegates too much to NModbus, it may lose timing, raw payload fidelity, malformed-frame diagnostics, or partial-frame behavior.

Recommended stance:

Use a hybrid model. MITMI owns the proxy/capture envelope. NModbus assists the Modbus plugin where it adds value.

## 9. Proposed Project Boundaries

The eventual project structure should separate responsibilities clearly.

Recommended conceptual projects:

- `Mitmi.Domain`: protocol-neutral domain objects and decisions.
- `Mitmi.Application`: session orchestration and use cases.
- `Mitmi.Abstractions`: plugin and infrastructure contracts if a separate contracts assembly becomes useful.
- `Mitmi.Infrastructure`: file storage, logging adapters, metrics adapters, configuration adapters.
- `Mitmi.Protocols.Modbus`: Modbus TCP plugin and NModbus integration.
- `Mitmi.Host.Console`: CLI and console host.
- `Mitmi.Tests.*`: focused tests by layer.

Challenge:

Too many projects too early create ceremony. Too few projects make boundary violations easy.

Recommendation:

Start with clear projects for host, application/core, and Modbus plugin when implementation begins. Split infrastructure and abstractions further only when the seams become real.

## 10. Middleware Pipeline For v0.1

v0.1 should keep the pipeline simple.

Required pipeline responsibilities:

- Observe message.
- Attach protocol metadata when available.
- Emit structured log event.
- Emit capture record.
- Emit metrics.
- Forward unchanged.

The forwarding path is latency-sensitive. Middleware should treat logging, capture, metrics, and semantic decoding as diagnostic side effects rather than prerequisites for forwarding whenever possible.

Not required in v0.1:

- Modify message.
- Block message.
- Delay message.
- Replay instead of forwarding.
- Cache response.
- Queue writes.

Design caution:

If v0.1 pipeline only supports pass-through logging, v0.2 may require a painful redesign for policy decisions. The pipeline should therefore return an explicit decision even when the only enabled decision is "forward unchanged".

Second caution:

Asynchronous diagnostics need bounded behavior. If log, capture, or metrics queues fall behind, MITMI must have a visible pressure/loss policy rather than growing memory without limit.

Recommended v0.1 decisions:

- Forward unchanged.
- Forward with warning metadata.
- Drop only on transport failure or explicit fatal configuration/runtime error.

Future decisions:

- Delay.
- Block.
- Replace.
- Replay.
- Cache.
- Queue.
- Disconnect.

## 11. Capture And Logging

v0.1 should distinguish logs from captures.

Logs are for humans and operations:

- Startup events.
- Configuration validation.
- Connection lifecycle.
- Request/response summaries.
- Warnings.
- Errors.
- Session summary.

v0.1 should log to both console and file by default.

Console and file logs should have independently configurable minimum levels:

- `Debug`
- `Info`
- `Warning`
- `Error`

Recommended defaults:

- Console: `Info`
- File: `Info`

`Debug` should be opt-in. It is useful during development and deep field diagnosis, but it can produce high-volume output and may expose sensitive operational details.

Console logs should prioritize readability and current operator awareness. File logs should prioritize durable diagnosis and supportability.

Captures are for later tooling and replay:

- Direction.
- Timestamp.
- Session ID.
- Connection ID.
- Raw payload.
- Protocol ID.
- Correlation ID when available.
- Modbus metadata when available.
- Decode warnings.

Captures should be enabled by default in v0.1.

When capture is enabled, startup diagnostics must show:

- Capture status.
- Resolved capture output path.
- Retention warning.
- Sensitive data warning.

Capture should be configurable so users can disable it explicitly or choose a different output path.

Trade-off:

One file format is simpler, but it encourages mixing operational logs with replay data. That becomes messy quickly.

Recommendation:

Use separate conceptual sinks from the start, even if both are initially file-based.

Do not put raw traffic payloads in normal `Info` logs. Raw payloads belong in capture records. If future `Debug` logging includes protocol details, it should be explicit and documented.

## 12. Configuration Direction

v0.1 configuration should be JSON and single-session first.

The configuration should describe:

- Application logging.
- Metrics output.
- Capture output.
- One session.
- For that session: protocol, listen endpoint, upstream endpoint, and diagnostic options.

Recommendation:

Keep configuration declarative. Do not add scripting or rule expressions in v0.1.

Risk:

If configuration is too abstract early, field users will struggle. If it is too Modbus-specific, later protocols will inherit awkward names.

Recommended compromise:

Use protocol-neutral session fields for transport and lifecycle, with a protocol-specific options section inside the session.

Recommended v0.1 shape:

- Configuration version.
- Application-level logging, metrics, and capture settings.
- A single `session` object.

Logging configuration should include separate console and file sections with independent minimum levels.

Capture configuration should include enabled/disabled status and output path. Retention should be visible in configuration even if v0.1 only supports manual cleanup warnings.

Session configuration should clearly distinguish the local listen endpoint from the upstream target endpoint.

Avoid a `sessions` array in v0.1. It adds ceremony before multi-session operation exists. The future migration to multiple sessions should be handled through configuration versioning.

## 13. Reliability And Safety Expectations

Because v0.1 is in the communication path, it must fail predictably.

Minimum expectations:

- Clear startup validation before listening.
- No silent fallback to unsafe defaults.
- Connection lifecycle logging.
- Upstream connection failure reporting.
- Listen-port binding error reporting.
- Firewall and permission hints where practical.
- Clear message that clients must be configured to connect to MITMI's listen endpoint.
- Graceful shutdown.
- Capture flush on shutdown.
- Clear capture output path at startup.
- Retention and sensitive data warning when capture is enabled.
- Bounded memory usage strategy.
- No active traffic modification in v0.1.

Challenge:

A diagnostic proxy that changes timing too much can create the problem it is trying to diagnose.

Recommendation:

Treat latency and forwarding errors as first-class metrics from the beginning. Prefer forwarding continuity over diagnostic completeness in the default v0.1 mode.

## 14. Performance Position

Performance matters, but maintainability still wins in v0.1.

Recommended targets should be practical rather than heroic:

- Support typical PLC/SCADA polling rates.
- Avoid unnecessary large allocations in the forwarding path.
- Preserve raw frames without excessive copying where reasonable.
- Prefer async I/O.
- Record latency metrics.
- Keep file writes and other slow diagnostics out of the hot forwarding path where practical.

Do not optimize for:

- Massive broker-scale throughput.
- Multi-protocol bridging performance.
- Distributed deployment.
- High-cardinality telemetry systems.

## 15. Testing Strategy For Later Implementation

When implementation starts, tests should prove:

- Configuration validation catches bad endpoints and missing sections.
- Modbus TCP frames can be identified and correlated.
- Raw forwarding still works when decode fails.
- Forwarding does not depend on log, capture, or metrics writes in the normal path.
- Logs and captures receive expected records.
- Diagnostic queue pressure or loss is visible if it occurs.
- Upstream disconnects are handled predictably.
- Shutdown flushes capture data.

Integration tests should eventually use:

- A local Modbus TCP server.
- A local Modbus TCP client.
- MITMI between them.

## 16. Risks And Mitigations

### Risk: MITMI Alters System Behavior

Mitigation: no modification features in v0.1, explicit pass-through mode, latency metrics, and clear connection logs.

### Risk: Modbus Leaks Into Core

Mitigation: Modbus metadata is plugin-provided and optional. Core owns envelopes and decisions, not function codes.

### Risk: NModbus Does Not Fit Proxy Needs

Mitigation: keep raw frame handling MITMI-owned and isolate NModbus inside the Modbus plugin.

### Risk: "Observe" Is Misunderstood As Passive Sniffing

Mitigation: document v0.1 as in-path diagnostic proxy. Use "pass-through diagnostic proxy" in product language.

### Risk: Capture Format Locks In Too Early

Mitigation: define capture concepts first. Treat the first file format as versioned and experimental.

### Risk: Too Much Architecture Before Working Software

Mitigation: keep v0.1 implementation slices small once coding begins: config validation, TCP pass-through, frame detection, logging, capture.

### Risk: Laptop Environment Disrupts Sessions

Mitigation: document firewall, sleep, VPN, adapter switching, and port-binding risks. Report listen/upstream endpoints clearly at startup.

### Risk: Users Expect Transparent Interception

Mitigation: clearly document that v0.1 requires explicit client retargeting to MITMI's listen endpoint. Avoid product language that suggests passive or transparent interception.

### Risk: Diagnostics Distort Traffic Timing

Mitigation: prefer latency-first forwarding, gather diagnostics asynchronously, and report diagnostic pressure or loss explicitly.

## 17. Recommended Implementation Slices When Approved

No code should be written until implementation is explicitly approved.

Recommended slices:

1. Create core/host project structure and configuration model.
2. Implement one session with TCP pass-through and graceful shutdown.
3. Add Modbus TCP frame detection and transaction correlation.
4. Add structured logging.
5. Add capture sink.
6. Add basic metrics.
7. Add integration test with local Modbus client/server.

Do not start with replay. Replay should wait until capture records are trustworthy.

## 18. Open Questions Before Implementation

No major v0.1 architecture questions are currently open. The next step should be implementation planning, including project structure, configuration shape, and the first runnable slice.
